# 办公小浣熊核心 Prompt（OPC 高手创造赛 · 可复用资产）

> 对应 **合同初审持续运营工作流** 的定时任务、Agent、联网、知识库、数据分析、PPT 等关键能力，可直接粘贴至办公小浣熊网页端。  
> 工作流说明见 [`docs/OPC_SUBMISSION.md`](../docs/OPC_SUBMISSION.md)。

---

## 模块 0 · 定时任务（每日/每周流水线）

### 0.1 创建每日 09:00 任务

```
请帮我创建一个办公小浣熊定时任务：

【任务名】合同初审-每日拉取
【调度】每个工作日 09:00（请给出 cron 表达式）
【执行内容】
1. 提醒我从 OA 待签目录 / 邮箱附件 / 共享盘拉取「昨日新增待审合同」
2. 对每份合同依次执行：文档解析 → 联网尽调乙方 → @知识库复核 → 生成审核报告
3. 高风险项汇总并提醒法务复核

请输出：cron 表达式、任务描述、失败重试建议（如 09:30 补跑一次）。
```

### 0.2 创建每周五 17:00 任务

```
请帮我创建一个办公小浣熊定时任务：

【任务名】合同初审-周报与法规巡检
【调度】每周五 17:00
【执行内容】
1. 联网检索本周增值税率、采购/服务合同相关法规是否有更新
2. 若有更新，生成「知识库红线补丁」文案供我粘贴入库
3. 汇总本周全部合同审核 JSON，做数据分析（通过率趋势、Top5 风险条款）
4. 按 docs/WEEKLY_BRIEFING_PPT.md 结构生成 6 页周报 PPT 大纲

请输出 cron 表达式与分步 Checklist。
```

## 模块 1 · 文档处理（合同解析）

```
你是企业法务助手。请阅读附件合同全文，抽取并输出 JSON：
- 合同名称、甲乙双方、合同金额（数字+大写）、税率、税额、付款比例、签效期
- 争议解决条款原文
- 保密条款原文
不要编造，缺失字段标为 null。
```

## 模块 2 · 知识库（@知识库 复核 · 读取）

```

对以下合同争议解决、保密、知识产权条款做语义冲突检测，
输出表格：条款摘要 | 风险等级 | 冲突点 | 推荐修改话术 | 知识库条目编号。

### 2.2 知识库 · 写入（沉淀案例）

```
请将本次合同《{file_name}》审核结论摘要写入知识库「合同风险案例库」，格式：
公司名 | 合同类型 | 高风险项（≤3条）| 处理结论 | 日期
不要重复已有条目，若乙方已在「对方主体画像库」请交叉引用。
```

## 模块 3 · 联网检索（对方主体尽调）

```
请开启办公小浣熊联网检索，查询合同乙方「{party_b}」：
1) 工商登记状态（存续/注销/异常）
2) 失信被执行人、经营异常名录
3) 近 12 个月涉诉案件与负面新闻

输出：
- 综合风险等级（高/中/低）
- 关键事实 bullet（每条附来源链接）
- 一段可写入知识库「对方主体画像库」的摘要（≤200字）

若信息不足，明确标注「需人工复核」，不要编造。
```

### 4.2 周报汇总（每周五）

```

1) 整体通过率 vs 上周（若未知则只报本周）

4) 累计节省人工小时数（假设人工 4h/份，AI 3min/份）
5) 3 条下周行动建议

输出 Markdown 表格，数值可从 JSON 计算，不得编造合同不存在的内容。

【JSON 数组】
{weekly_audit_json_array}

## 模块 6 · 汇报/PPT

### 6.1 单份合同 · 6 页汇报大纲

```
请把以下合同审核结论做成 6 页汇报 PPT 大纲（商务风格）：
- 封面：合同名 + 合规通过率
- 背景：人工初审痛点（4小时/份）
- 数据：金额/税率/付款比例
- 总览：22 项规则统计
- 风险：Top 5 不通过项
- 决策：修订后复审 / 附条件签署

每页不超过 5 个要点，可直接用办公小浣熊 PPT 生成。
主色建议：深蓝 + 铜色点缀。
```

每页 ≤5 要点，适合粘贴至办公小浣熊 PPT 一键生成。

## Agent 编排 · 单份全链路（一次对话）

```
你是办公小浣熊合同初审 Agent。请对附件合同按顺序执行，每步简短汇报进度：

1. 【文档处理】抽取关键字段 JSON
2. 【联网检索】尽调乙方主体风险（开启联网）
3. 【知识库】@contract_audit_kb 做红线冲突检测
4. 【数据分析】基于 22 条规则结果输出通过率与风险分布（若我有 JSON 则直接用）
5. 【文案】生成 Markdown 审核报告
6. 【知识库】将结论摘要写入案例库格式
7. 【汇报】输出 6 页 PPT 大纲

规则：不得编造；计算类结论以结构化 JSON 为准；每步输出控制在可读长度，最后给「高风险清单」。
```
